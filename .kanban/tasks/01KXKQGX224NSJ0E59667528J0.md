## What
`Tests/FoundationModelsCodeContextTests/ConnectionTests.swift:317`'s test `aRequestTimeoutThatPropagatesUncaughtStillClosesTheConnectionAndReapsTheProcess()` intermittently fails with:

```
Expectation failed: kill(capturedPID, 0) == -1 && errno == ESRCH
```

Observed failing once in 4 consecutive `swift test` runs (2026-07-15), passing the other 3 times. Discovered incidentally while verifying task 01KXK6WVQ6584D77AK33NFTMJZ (unrelated Examples/ManagerExample addition) — this test file was not touched by that change.

The assertion checks that a killed/reaped subprocess's PID no longer exists (`kill(pid, 0)` returns -1 with `ESRCH`). This is inherently a race against the OS actually finishing reaping the process; under load or scheduler jitter the check can run before the process is fully gone.

## Root cause
`ProcessLanguageServerConnection.close()` sends `SIGKILL` to the child and returns immediately, without waiting for the kernel/Foundation to actually finish reaping the process. A caller checking `kill(pid, 0) == -1 && errno == ESRCH` immediately after `close()` returns can race that asynchronous reap.

An attempted production fix (`process.waitUntilExit()` inside `close()` right after the `kill(SIGKILL)`) was tried and reverted: it deadlocked (confirmed via `sample` on the hung process, stuck in `-[NSConcreteTask waitUntilExit]` called from `close()`). `close()` is unchanged from its original form.

## Fix
Test-only fix in `ConnectionTests.swift`: added a `pidExists(_:)` helper and replaced the single immediate assertion with a bounded poll (100 x 10ms = 1s budget), mirroring the existing sibling pattern in `waitUntilExitReturnsOnceTheScriptRunsOut()`.
